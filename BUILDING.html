How to build VirtualGL in 9043 easy steps

Prerequisites - Linux

OpenSSL headers and link libraries

  • On most Linux variants, these can be found in the openssl-devel package included with the distribution. If your Linux distribution does not contain a pre-packaged version of OpenSSL, then you will need to build and install the toolkit from source. The source code can be obtained from http://www.openssl.org/.

GCC and G++ (tested with v2.96 and v3.x)

X11 dev. kit

  • On most Linux variants, this can be found in the XFree86-devel or xorg-x11-devel package.

GNU Make v3.7 or later

TurboJPEG

  • Download and install the appropriate TurboJPEG binary for your Linux distribution. TurboJPEG binaries can be found at https://sourceforge.net/project/showfiles.php?group_id=117509&package_id=166100. These binaries should work with all GLIBC 2.2 and newer systems.
  • Normally, you should not need to rebuild TurboJPEG from source. But if you do, see the notes below.

Prerequisites - x86-64 Linux

Building a redistributable VirtualGL package for x86-64 requires building both 32-bit and 64-bit versions of the VirtualGL server components. But not all x86-64 Linux systems contain the 32-bit packages necessary to build a 32-bit version of VirtualGL. To build 32-bit VirtualGL on an x86-64 Linux system, you will need at minimum the following 32-bit packages:

  • glibc development libraries
    • Usually contained in the glibc-devel package (iX86)
    • Usually installed by default on x86-64 Linux systems
  • X11 dev. kit
    • Usually contained in the XFree86-devel or xorg-x11-devel package (iX86)
    • Not always installed by default but usually available on the installation discs
  • OpenSSL runtime
    • Usually contained in the openssl package (iX86)
    • Not always installed by default but usually available on the installation discs
    • May also require 32-bit krb5-libs package

If for some reason your preferred 64-bit Linux distribution does not contain 32-bit packages for the above, then:

  1. Complain to your distribution vendor. It's ridiculous to not have the ability to cross-compile 32-bit apps on x86-64 systems.
  2. All x86-64 distributions of Linux have a corresponding 32-bit-only distribution, usually released on a separate set of discs. On those discs, you should be able to find the 32-bit versions of OpenSSL, the X11 dev. kit, and the GLIBC dev. kit to match the 64-bit versions that are already installed on your system. You may have to use rpm --force to install the 32-bit development packages, since they might contain header files which conflict with the 64-bit development packages that are already installed. Again, complain to your distribution vendor.

When you attempt to build 32-bit VirtualGL on a 64-bit Linux system, the VirtualGL build system will run a script (sanitycheck64, in the same directory as this document) to sanity check the 32-bit build environment on your system. This script points out any anomalies that would prevent 32-bit VirtualGL from building properly and gives instructions on how to fix them. You can also run sanitycheck64 -fix as root to automatically fix the problems. View the contents of the script if you're morbidly curious about what it's doing.

Rebuilding TurboJPEG on Linux

TurboJPEG is a simple but fast JPEG codec that VirtualGL uses as its default method of image compression. TurboJPEG abstracts a variety of hardware-specific JPEG compression technologies, some of them closed source, into a single API that VirtualGL can use across all of the platforms it supports. The Linux version of TurboJPEG currently relies on the Intel Integrated Performance Primitives (IPP), a proprietary set of libraries, to achieve the levels of performance necessary to support VirtualGL. Normally, the TurboJPEG library is built as part of the VirtualGL build process, but on Linux it was necessary to split TurboJPEG into a separate RPM so that end users could easily rebuild VirtualGL from source when needed (such as to install on a Linux distribution for which no VirtualGL binaries are available.)

The source code for TurboJPEG is contained within the VirtualGL build tree under the jpeg/ directory. To rebuild the TurboJPEG library: 

cd jpeg
make lib
You can then rebuild the TurboJPEG RPM by typing 
make dist
Rebuilding TurboJPEG from source requires one of the following:

  • Intel Integrated Performance Primitives (IPP)

    Single developer licenses for IPP can be obtained directly from Intel ( http://www.intel.com/software/products/ipp/index.htm.) Licenses are free for non-commercial use and inexpensive for commercial use. The IPP license entitles the license holder to distribute any binaries built with IPP freely and without royalties.

    Building TurboJPEG against IPP is the default if the JPEGLIB makefile option is not specified.
     
  • Pegasus Imaging PicTools RapidVue

    Passing an argument of JPEGLIB=pegasus to the makefile will build TurboJPEG against the PicTools RapidVue codec from Pegasus Imaging (http://www.jpg.com.)

    make JPEGLIB=pegasus lib
    make JPEGLIB=pegasus dist

    The TurboJPEG build environment looks for PICTools in a directory called pictools located one level up from the top directory of the VirtualGL source tree. You can override this by passing an additional argument of PEGDIR={location of pictools directory} to the makefile.
     
  • libjpeg v6b

    TurboJPEG can be built against a completely free (as in freedom) version of libjpeg v6b which will produce correct, albeit slow, behavior. Pass an argument of JPEGLIB=libjpeg to the makefile to build a version of TurboJPEG based on 100% open source technology.

    make JPEGLIB=libjpeg lib
    make JPEGLIB=libjpeg dist

    The custom libjpeg code is included in the jpeg/jpeg-6b directory of the VGL build and differs from the IJG release (see http://www.ijg.org) only in its support of little endian and 32-bit pixel formats (both of which are necessary to allow for decompressing directly into X11 shared memory.)

TurboJPEG Makefile Options

Each of these options can be passed as an argument to "make".

IPPDIR = {location of Intel Performance Primitives directory}

This allows you to override the default location of /opt/intel/ipp/5.1/{ia32|em64t|itanium}, for instance to use an earlier or later version of IPP.

IPPSHARED = yes
Link against the IPP shared libraries rather than the static (merged) libraries. Using this option has the following ramifications:
  • The IPP run-time RPM must be installed on the build machine.
  • The IPP run-time RPM must be installed on any machine that will use TurboJPEG.
  • TurboJPEG's CPU autodetection mechanism for non-Intel CPU's will no longer work, and it is likely that the non-optimal (MMX) code path will be selected for non-Intel CPU's at run time.
See the "Makefile Options" section below for other makefile options that may affect the TurboJPEG build.

Prerequisites - Solaris

Solaris 8 or newer (Sparc) or Solaris/OpenSolaris 10 or newer (x86)

OpenSSL headers and link libraries

  • If present, VirtualGL will use the following dynamically linked versions of OpenSSL (in the following order) to satisfy compile time and run time dependencies:

    • Blastwave (http://www.blastwave.org) version of OpenSSL (contained in /opt/csw) -- recommended for Solaris 8 and 9 systems.
    • Solaris 10 version of OpenSSL (contained in /usr/sfw.) VirtualGL tries to use this if Blastwave is not installed.

    Building against either version of OpenSSL will introduce run-time dependencies on libssl.so and libcrypto.so, so these dynamic libraries must be present on any system that will run VirtualGL in SSL mode. They need not be present if SSL mode is not used. Thus, any Solaris machine (client or server) that will use VirtualGL's SSL mode of operation will need to have either Solaris 10 OpenSSL or Blastwave OpenSSL installed.

  • If you so choose, you can also create a custom build of OpenSSL using the source code obtained from http://www.openssl.org/. Untar the source, then move the openssl build directory to openssl.solaris (for a 32-bit Sparc build), openssl.solaris64 (for a 64-bit Sparc build), openssl.solx86 (for an x86 build), or openssl.solx8664 (for an x86-64 build.) VirtualGL will look for this directory one level up from its own build directory, for example: 
    ls -l ~/src
    drwx------ 12 drc staff Jan 20 12:17 .
    drwx------ 59 drc staff Jan 20 13:43 ..
    drwx------ 21 drc staff Dec 20 12:17 openssl.solaris
    drwx------ 21 drc staff Dec 13 13:40 openssl.solaris64
    drwx------ 21 drc staff Feb 12 11:03 openssl.solx86
    drwx------ 21 drc staff Feb 12 11:08 openssl.solx8664
    drwx------ 14 drc staff Jan 20 12:17 vgl
    CD to the appropriate directory and build OpenSSL using the instructions provided in the OpenSSL source distribution.

    Note: To configure the OpenSSL build for a specific processor type and compiler, refer to the following table:

    Sun Studio/Workshop Pro GCC
    Sparc (32-bit) ./config ./Configure solaris-sparcv9-gcc shared
    Sparc (64-bit) ./Configure solaris64-sparcv9-cc ./Configure solaris64-sparcv9-gcc shared
    x86 ./Configure solaris-x86-cc ./Configure solaris-x86-gcc shared
    x86-64 ./config ./Configure solaris64-x86_64-gcc shared

    Using the "shared" flag with GCC is necessary in order to build OpenSSL with position-independent code, but it has the side effect of generating both a static and dynamic library. If you would prefer to link OpenSSL statically, just delete libssl.so and libcrypto.so from your OpenSSL build directory prior to building VirtualGL.

Building

Assuming all of the above is set up correctly, you should be able to just type 

make

On Solaris, type

gmake

instead.

Makefile Options

Each of these options can be passed as an argument to make or gmake.

DEBUG = yes

Build a version of VirtualGL with debugging symbols and no optimization. The debug version will reside in ${platform}/dbg/lib and ${platform}/dbg/bin, whereas the release version will reside in ${platform}/lib and ${platform}/bin, so these two versions can be built independently of one another.

JPEGLIB = < libjpeg | pegasus >

libjpeg = Build VirtualGL using a custom version of libjpeg v6b. This custom libjpeg code is included in the jpeg/jpeg-6b directory and differs from the IJG release (see http://www.ijg.org) only in its support of little endian and 32-bit pixel formats (both of which are necessary to allow for decompressing directly into X11 shared memory.)

pegasus = Use the PICTools RapidVue codec from Pegasus Imaging (http://www.jpg.com.) The VirtualGL build environment looks for PICTools in a directory called pictools located one level up from the build directory. You can override this by passing an additional argument of PEGDIR={location of pictools directory}.

M32 = yes

On x86-64 systems, this cross-compiles a 32-bit (x86) version of VirtualGL.

CC = gcc

On Solaris, this builds VirtualGL using the GCC compiler rather than the default (Sun WS Pro) compiler. gcc and g++ should be in your PATH.

USEGLP = no

On Solaris, this builds VirtualGL without GLP extension code. This is necessary to build VirtualGL on versions of Sun OpenGL prior to 1.5.

USESSL = no

Build VirtualGL without SSL support.

Installing / Distributing

On Unix systems, you can type 

make install [prefix={install prefix}]

prefix defaults to /usr/local but can be changed to something else (e.g. /usr) if you prefer. Executables will be installed to $prefix/bin, and libraries will be installed to $prefix/lib (or $prefix/lib/sparcv9 or $prefix/lib/64 or $prefix/lib64, as appropriate.)

To build a redistributable package, type 

make dist
  • On Linux, this requires RPM v4 or later.
  • On Solaris, this requires pkgmk, pkgtrans, and bzip2.
  • On Windows, this requires the Nullsoft Install System ( http://nsis.sourceforge.net/.) makensis.exe should be in your PATH.
